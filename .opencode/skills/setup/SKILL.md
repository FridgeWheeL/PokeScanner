# Setup Skill

Generates a tailored OpenCode configuration based on your project's
actual stack, structure, and needs. Works with .NET, Terraform, Node,
Python, mono-repos, and any other project type.

Templates live under `.opencode/skills/setup/templates/`.
Each file contains `{{PLACEHOLDER}}` markers that get substituted.

## Phase 1: Deep scan

Run these scans to auto-detect as much as possible. Record all results.

### Solution type detection

Determine the primary project type(s) by checking for characteristic files:

| File/directory | Likely type |
|----------------|-------------|
| `*.sln` or `*.csproj` | .NET solution |
| `*.tf` or `*.tfvars` | Terraform |
| `package.json` | Node.js |
| `pyproject.toml`, `requirements.txt`, `setup.py` | Python |
| `Cargo.toml` | Rust |
| `go.mod` | Go |
| `pom.xml`, `build.gradle` | Java |

Record `SOLUTION_TYPE` (comma-separated for mixed repos).

### Project role detection (.NET specific)

For each `.csproj` or project directory, determine its role:

| Heuristic | Role |
|-----------|------|
| Contains `Worker` or `BackgroundService` | Worker / Background service |
| Contains `Api`, `Web`, `Mvc`, `Blazor` | API / Web (presentation layer) |
| References `MSTest.TestAdapter`, `xunit`, `NUnit` | Test project |
| Contains `Domain` or `Core` in name | Domain / Core library |
| Contains `Infrastructure` or `Data` or `Persistence` | Infrastructure / Data |
| Contains `Application` or `UseCases` or `Services` | Application / Use-cases |
| Contains `Integration` or `Tests.Integration` | Integration test project |
| Contains `Console` | Console app |
| Contains `Function` or `Functions` | Azure Function / Lambda |
| None of the above | Shared library / Utilities |

Record each project as `PROJECT_1_NAME`, `PROJECT_1_ROLE`, etc.

### Framework detection

For .NET projects, scan each non-test `.csproj` for packages:

| Category | Packages to check |
|----------|-------------------|
| Test SDK | `MSTest.TestAdapter`, `xunit.runner.visualstudio`, `NUnit3TestAdapter` |
| Mock lib | `NSubstitute`, `Moq`, `FakeItEasy` |
| ORM | `Microsoft.EntityFrameworkCore`, `Dapper`, `NHibernate` |
| Web framework | `Microsoft.AspNetCore.App`, `Swashbuckle`, `FastEndpoints` |
| Logging | `Serilog`, `NLog`, `log4net` |
| Mapping | `AutoMapper`, `Mapster` |
| CQRS | `MediatR` |
| Validation | `FluentValidation` |

For Node projects, scan `package.json` for relevant frameworks
(Express, Next.js, NestJS, Jest, Vitest, etc.).

For Python projects, scan `pyproject.toml` or `requirements.txt`
(Django, FastAPI, Flask, pytest, etc.).

For Terraform projects, scan `*.tf` for provider references.

### Structure analysis

- Check for `src/` and `tests/` directories
- Check for existing `.opencode/`, `opencode.json`, `AGENTS.md`
- Check `.editorconfig` for `indent_style`, `indent_size`, `charset`
- Check `.gitignore` and `.gitattributes`
- Check `.github/workflows/`, `azure-pipelines.yml`, `.gitlab-ci.yml`

### Existing config scanning

If `.opencode/` or `AGENTS.md` or `opencode.json` already exist, scan
them thoroughly for consolidation:

- Glob `.opencode/agent/*.md` → read each file's frontmatter and body
- Read existing `AGENTS.md` fully
- Read existing `opencode.json`
- Read existing `.opencode/skills/*/SKILL.md` files

For each existing file, record `file_path`, `frontmatter`, `body`,
and `size`. Check whether files were generated by a previous setup
run or hand-written.

### Model discovery

Scan for available LLM models to present as options during setup:

1. Read existing `opencode.json` and extract:
   - The default `model` field if set.
   - All `provider.*.models` entries — collect every model ID from
     every configured provider.
   - Any per-agent `model` overrides (these indicate models the user
     already has access to).
2. Check environment variables for provider API keys:
   - `ANTHROPIC_API_KEY` → Anthropic models available
   - `OPENAI_API_KEY` → OpenAI models available  
   - `GOOGLE_API_KEY` or `GEMINI_API_KEY` → Google models available
   - `GROQ_API_KEY` → Groq models available
   - Any other `*_API_KEY` or `*_API_KEY` env vars with non-empty values
3. Build `AVAILABLE_MODELS` — a deduplicated list of all discovered
   model IDs, plus well-known models for each detected provider.
   Format each as `provider/model` (e.g., `anthropic/claude-sonnet-4-20250514`).

Well-known models by provider:

| Provider | Models |
|----------|--------|
| anthropic | `claude-sonnet-4-20250514`, `claude-haiku-3-5-20241022`, `claude-opus-4-20250514` |
| openai | `gpt-4.1`, `gpt-4.1-mini`, `gpt-4.1-nano`, `o3`, `o4-mini` |
| google | `gemini-2.5-pro`, `gemini-2.5-flash` |
| groq | `llama-4-scout-17b-16e-instruct`, `llama-4-maverick-17b-128e-instruct` |
| github | `copilot/gpt-4.1`, `copilot/claude-sonnet-4-20250514` |
| deepseek | `deepseek/deepseek-chat`, `deepseek/deepseek-reasoner` |

Include `default` as a special entry meaning "use whatever model is
calling this agent" (i.e., inherit the caller's model).

### Code pattern sampling (.NET only)

Read 2-3 representative `.cs` files and detect:

| Pattern | What to look for |
|---------|-----------------|
| Nullable enabled | `#nullable enable` or `<Nullable>enable</Nullable>` in csproj |
| Namespace style | `namespace Foo.Bar;` (file-scoped) vs `namespace Foo.Bar { ... }` (block) |
| Private field style | `_camelCase` vs `_CamelCase` vs `camelCase` |
| Exception handling | `Result<T>` usage vs plain `throw` |
| Primary constructors | `class Foo(IBar bar)` syntax |

## Phase 2: Questions

Build questions dynamically based on what Phase 1 detected. Only ask
about what wasn't detected or needs confirmation.

### Solution identity (always ask)

1. "Project name: [{{detected}}] — correct?"
2. "Solution type: {{SOLUTION_TYPE}} — correct? Add/remove any?"

### Project roles (if .NET with multiple projects)

3. "Projects detected: [list each with role]. Correct? Any to add or
   reclassify?"

### Testing (only if test project detected)

4. "Test framework: MSTest / xUnit / NUnit / None (detected: {{det}})"
5. "Mock library: NSubstitute / Moq / FakeItEasy / None (det: {{det}})"

### Stack (only for .NET)

6. "ORM: Entity Framework Core / Dapper / NHibernate / None (detected: {{det}})"
7. "Architecture pattern: Clean Architecture / N-tier / Onion / None / Custom :"

### Cross-platform

8. "CI platform: GitHub Actions / Azure DevOps / GitLab CI / None (detected: {{det}})"
9. "Formatter command (e.g., `dotnet format`, `prettier`, `terraform fmt`, `ruff`):"
10. ".NET version (if applicable, from TFM e.g., `net8.0`):"
11. "Any custom conventions to add to AGENTS.md? (paste or describe)"

### LLM model assignment

These questions let you assign different models to different sub-agents.
Discovered models (from existing config, env vars, and well-known lists)
are presented as options. Type `default` to inherit the caller's model.

12. **Select a model for each sub-agent.** For each one, enter the full
    `provider/model` name, pick from the discovered list below, or enter
    `default` to use whatever model invokes that sub-agent.

    Discovered models: {{AVAILABLE_MODELS}}

    | Agent | Role | Recommended | Your model |
    |-------|------|-------------|------------|
    | planner | Designs implementation plans | High reasoning model | `{{MODEL_PLANNER}}` |
    | coding-agent | Writes production code | Medium reasoning model | `{{MODEL_CODING}}` |
    | test-specialist | Writes tests | Medium reasoning model | `{{MODEL_TEST}}` |
    | reviewer | Reviews code, read-only | Medium-high reasoning model | `{{MODEL_REVIEWER}}` |
    | cleanup-agent | Whitespace, polish, comments | Low/no reasoning, fast/cheap | `{{MODEL_CLEANUP}}` |
    | solutions-architect | Architecture decisions | High reasoning model | `{{MODEL_ARCHITECT}}` |

    Enter `default` for any agent that should inherit the default model.
    Enter a custom `provider/model` value (e.g., `anthropic/claude-sonnet-4-20250514`)
    to pin a specific model.

### Extras

14. "Install the grill-me skill from mattpocock/skills? It relentlessly
    interviews you about a plan/design until every branch of the decision
    tree is resolved — helps align you and the AI before coding. [Y/n]"

### Existing config handling

If any existing OpenCode files were detected, handle them per-file:

**For each detected existing file**, ask:

1. "Existing file found: `{path}`. What would you like to do?"
   - **Overwrite** — Replace entirely with the generated version
   - **Skip** — Keep the existing file as-is
   - **Consolidate** — Read existing, merge with template, ask
     about conflicts

2. If **Consolidate**:
   - Compare existing frontmatter/body against the template.
   - For each differing section, present the conflict:
     ```
     Conflict in {path} ({section}):
       Existing: {existing_value}
       Template: {template_value}
     Which one to keep? [existing / template / edit]
     ```
   - For new sections: "Add new section '{section}' from template?"
   - For existing-only sections: "Keep custom section '{section}'?"
   - Allow free-form edits when user picks "edit".

3. Collect all decisions into a merge plan.

```
PROJECT_1_NAME=MyProject.Api
PROJECT_1_ROLE=api
PROJECT_2_NAME=MyProject.Application
PROJECT_2_ROLE=application
```

### Model assignment (per-agent LLM models)

User-facing values (raw model names, e.g., `anthropic/claude-sonnet-4-20250514`):

```
MODEL_CODING=
MODEL_TEST=
MODEL_REVIEWER=
MODEL_CLEANUP=

```

Leave empty or enter `default` to use whatever model invokes the
sub-agent (inherits the caller's model). Enter a specific

```
MODEL_ARCHITECT_LINE=
```

corresponding `MODEL_*` has a value, or empty string when it's blank.

## Phase 4: Build section flags

After Phases 1-3, set the section flags to determine which content
appears in AGENTS.md and which agents are created.

### Coding standards

Set `SECTION_CODING_STANDARDS` based on solution type:

| Solution type | Content |
|---------------|---------|
| dotnet (with .cs files) | C# coding standards (naming, XML docs, null handling, DI, etc.) |
| node | TypeScript/JS standards (ESLint, Prettier, naming conventions) |
| python | Python standards (PEP 8, type hints, naming) |
| terraform | HCL standards (formatting, naming, module structure) |
| terraform + dotnet | Combined: C# standards for .NET code |
| unknown | Generic file and git conventions only |

If `HAS_CODE_STANDARDS=false`, set `SECTION_CODING_STANDARDS` to empty.

### Testing section

Set `SECTION_TESTING` only if `HAS_UNIT_TESTS=true`:

- For dotnet: include testing framework conventions and mock library
- For node: include JS/TS testing framework conventions
- For python: include pytest conventions
- For terraform: skip (no testing section)

### Architecture section

Set `SECTION_ARCHITECTURE` if:
- A .NET architecture pattern was selected (Clean Architecture, etc.)
- An architecture skill was requested
- Multiple project roles exist

Otherwise, skip it.

### Stack entries

Build `{{STACK_ENTRIES}}` based on detected technologies:

- For .NET: add Runtime, Presentation, Application, Infrastructure,
  and Testing table sections with detected packages.
  Set `DOTNET_VERSION_ROW` to the `DOTNET_VERSION` value (e.g. `net8.0`).
- For other platforms, leave `DOTNET_VERSION_ROW` as empty string.
- For other platforms: add relevant table sections per detected tech.
- If no specific tech detected, leave it as the generic "Project" entry
  that's already in the template.

### Architecture diagram & dependency rules

Build `{{ARCHITECTURE_DIAGRAM}}` and `{{DEPENDENCY_RULES}}` based on
the architecture pattern:

- For Clean Architecture / Onion / Hexagonal: include the layered
  mermaid diagram with Domain, Application, Infrastructure, Presentation.
- For N-tier: include a simpler tier-based diagram.
- For Terraform: include a module dependency graph.
- For monolith with no pattern: include a basic component diagram.

Build `{{PROJECT_ROLES_TABLE}}` from the detected project roles:

```markdown
| Project | Role |
|---------|------|
| MyProject.Api | API / Web (presentation layer) |
| MyProject.Application | Application / Use-cases |
| MyProject.Domain | Domain / Core library |
| MyProject.Infrastructure | Infrastructure / Data |
| MyProject.Tests | Test project |
```

If no projects were detected, omit the table.

### Model line computation

For each `{{MODEL_*}}` value from the substitution map, compute the
corresponding `{{MODEL_*_LINE}}`:

| Raw value (MODEL_X) | Computed line (MODEL_X_LINE) |
|---------------------|------------------------------|
| `anthropic/claude-sonnet-4-20250514` | `"model": "anthropic/claude-sonnet-4-20250514",` |
| `default` | (empty string) |
| (empty) | (empty string) |

When `MODEL_X` is empty or `default`, the `MODEL_X_LINE` is empty,
meaning no `"model"` key appears in opencode.json for that agent.
The sub-agent inherits the model used by whatever called it.

### Model selection guidance

Different sub-agents benefit from different model capabilities:

| Agent type | Reasoning level | Why | Example models |
|------------|----------------|-----|----------------|
| **planner** | High | Needs to reason about architecture trade-offs, sequence dependencies, and edge cases | `claude-sonnet-4-20250514`, `claude-opus-4-20250514`, `gpt-4.1`, `gemini-2.5-pro` |
| **solutions-architect** | High | Makes irreversible structural decisions; needs deep system-level reasoning | `claude-opus-4-20250514`, `gpt-4.1`, `gemini-2.5-pro` |
| **coding-agent** | Medium | Follows detailed specs; needs good code generation but less strategic reasoning | `claude-sonnet-4-20250514`, `gpt-4.1-mini`, `gemini-2.5-flash` |
| **test-specialist** | Medium | Generates test cases; needs to understand code paths but follows established patterns | `claude-sonnet-4-20250514`, `gpt-4.1-mini`, `gemini-2.5-flash` |
| **reviewer** | Medium-high | Spots bugs, anti-patterns, and doc drift; needs a sharp eye but limited scope | `claude-sonnet-4-20250514`, `gpt-4.1`, `gemini-2.5-flash` |
| **cleanup-agent** | Low / none | Mechanical tasks: whitespace, imports, comment removal. Reasoning overhead is wasted | `claude-haiku-3-5-20241022`, `gpt-4.1-nano`, `gemini-2.5-flash` |

Key principle: **Don't use a reasoning-heavy model for mechanical tasks**.
The cleanup-agent, for example, does trivial pattern matching — a fast,
cheap model with no reasoning is optimal. Conversely, the planner and
solutions-architect benefit from models with strong reasoning because
their output shapes the entire implementation.

When choosing models:
- **High reasoning models** (`opus`, `sonnet`, `gpt-4.1`, `gemini-2.5-pro`):
  Use for planning, architecture, and complex code review
- **Medium reasoning models** (`sonnet`, `gpt-4.1-mini`, `gemini-2.5-flash`):
  Good for coding and test writing where the spec is clear
- **Low reasoning models** (`haiku`, `gpt-4.1-nano`): Perfect for cleanup,
  formatting, and simple edits

Format: always use `provider/model` syntax (e.g.,
`anthropic/claude-sonnet-4-20250514`).

## Phase 5: File generation

Templates are stored under `.opencode/skills/setup/templates/`.
For each template:

1. Determine the template file path from the table below.
2. Read the template file content.
3. Replace every `{{PLACEHOLDER}}` with the value from the substitution map.
4. If `{{CUSTOM_RULES}}` is non-empty, insert at the `{{CUSTOM_RULES}}`
   marker. If empty, remove the marker line and its preceding heading line
   and trailing blank line so no orphaned heading remains.
5. Replace `{{SECTION_CODING_STANDARDS}}`, `{{SECTION_TESTING}}`,
   `{{SECTION_ARCHITECTURE}}`, `{{STACK_ENTRIES}}`,
   `{{ARCHITECTURE_DIAGRAM}}`, `{{DEPENDENCY_RULES}}`,
   `{{PROJECT_ROLES_TABLE}}` with their computed content.
   When a `{{SECTION_*}}` placeholder is empty (i.e. the section should
   be omitted), also remove the preceding `## N. SectionName` heading
   line and its trailing blank line so no orphaned heading remains.
6. Replace all `{{MODEL_*_LINE}}` placeholders with their computed values
   (set in Phase 4).
6. If `{{GRILL_ME_ENABLED}}` is `true`:
   - Replace `{{GRILL_ME_STEP}}` with a `/grill-me` step before planning
   - Replace `{{GRILL_ME_PATHS_ENTRY}}` with `, ".agents/skills"`
   If `false`: remove `{{GRILL_ME_STEP}}` line, empty `{{GRILL_ME_PATHS_ENTRY}}`.
7. If a conditional file should be skipped (see conditions table), omit it.
8. Check if the output path exists and handle per user choice (Overwrite /
   Skip / Consolidate).
9. Write the result, creating parent directories as needed.

## Phase 6: Optional extras

### grill-me skill

If the user opted in for the grill-me skill (Phase 2, question 12):

1. Set `GRILL_ME_ENABLED=true` in the substitution map.
2. Run: `npx skills@latest add mattpocock/skills`
3. Inform user to select `/setup-matt-pocock-skills` during install,
   then run `/setup-matt-pocock-skills` in their agent.

Notify the user when done: "Setup complete. Restart OpenCode for the
new config to take effect."
